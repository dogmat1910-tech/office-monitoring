# ТЗ: Интеграция корпоративного календаря → office-monitoring через webhook

**Заказчик**: Егор (office-monitoring)
**Исполнитель**: команда разработки корпоративного календаря
**Дата**: 2026-05-26

---

## 1. Контекст и цель

Система **office-monitoring** контролирует работу менеджеров: ведёт учёт активного окна, скриншоты, аудио со встреч (через always-on микрофон с VAD-фильтром). LLM-анализ оценивает качество встреч менеджера с клиентом и формирует ежедневный отчёт.

Для корректного анализа сервер должен знать **когда у каждого менеджера началась и закончилась встреча с клиентом**, чтобы правильно сегментировать аудио. Сейчас встречи отмечаются вручную кнопкой в дашборде — это ненадёжно и трудозатратно.

**Цель интеграции** — корпоративный календарь сам уведомляет office-monitoring о статусе встреч, без участия менеджера.

API календаря закрытый (только внутренняя сеть), поэтому связь делается **в одну сторону**: календарь шлёт HTTP-запросы (webhook) на наш сервер. Мы запросы к календарю не делаем.

---

## 2. Что нужно реализовать на стороне календаря

При наступлении событий в календаре отправлять HTTP POST на наш endpoint:

```
POST https://office.lkdzrkk.pro/webhook/calendar
Authorization: Bearer <om_webhook_token>
Content-Type: application/json
```

**`om_webhook_token`** — длинная случайная строка, выдадим отдельно (хранится в .env вашей системы, в логи и репозитории не попадает).

---

## 3. События

### 3.1 Обязательные

| Event              | Когда отправлять                                                      |
|--------------------|------------------------------------------------------------------------|
| `meeting.started`  | Встреча перешла в активный статус (фактически началась)               |
| `meeting.ended`    | Встреча завершилась (нормально или досрочно)                          |

### 3.2 Желательные (повышают надёжность)

| Event                | Когда отправлять                                              |
|----------------------|---------------------------------------------------------------|
| `meeting.scheduled`  | За 5–10 минут до начала запланированной встречи                |
| `meeting.cancelled`  | Встреча отменена до начала                                     |
| `meeting.updated`    | Сдвиг времени, смена клиента или менеджера                     |

`meeting.scheduled` нужен как **подстраховка**: если `meeting.started` потеряется в сети, у нас всё равно есть запись об ожидающейся встрече, и аудио привяжется к ней по timestamp'ам.

---

## 4. Формат payload

```json
{
  "event": "meeting.started",
  "external_id": "calendar-meeting-uuid-abc-12345",
  "manager_email": "ivanov@company.ru",
  "manager_external_id": "USR-00451",
  "manager_name": "Иванов Иван Иванович",
  "started_at": "2026-05-26T14:30:00+03:00",
  "ended_at": null,
  "scheduled_started_at": "2026-05-26T14:30:00+03:00",
  "scheduled_ended_at": "2026-05-26T15:30:00+03:00",
  "client_name": "ООО Ромашка",
  "client_external_id": "CL-9912",
  "meeting_kind": "client_offline",
  "subject": "Презентация продукта",
  "location": "Офис Москва, переговорка 3"
}
```

### Описание полей

| Поле                    | Тип       | Обязательность | Описание                                                                 |
|-------------------------|-----------|----------------|--------------------------------------------------------------------------|
| `event`                 | string    | **обяз**       | `meeting.scheduled` / `meeting.started` / `meeting.ended` / `meeting.cancelled` / `meeting.updated` |
| `external_id`           | string    | **обяз**       | Стабильный ID встречи в вашей системе. По нему делаем идемпотентность.   |
| `manager_email`         | string    | желат.         | Email менеджера-организатора                                              |
| `manager_external_id`   | string    | желат.         | Ваш внутренний ID менеджера (стабильный, не меняется при переименовании)  |
| `manager_name`          | string    | желат.         | ФИО менеджера                                                             |
| `started_at`            | ISO8601   | для `started`  | Фактическое время начала с таймзоной                                      |
| `ended_at`              | ISO8601   | для `ended`    | Фактическое время окончания                                               |
| `scheduled_started_at`  | ISO8601   | желат.         | Плановое время начала из календаря                                        |
| `scheduled_ended_at`    | ISO8601   | желат.         | Плановое время окончания                                                  |
| `client_name`           | string    | желат.         | Название клиента/контрагента                                              |
| `client_external_id`    | string    | желат.         | Ваш ID клиента в CRM                                                      |
| `meeting_kind`          | enum      | желат.         | `client_offline` / `client_online` / `internal` / `personal` — определяет нужно ли вообще анализировать аудио |
| `subject`               | string    | желат.         | Тема встречи                                                              |
| `location`              | string    | желат.         | Место (офис/онлайн-комната)                                               |

### Идентификатор менеджера

**Достаточно одного** из трёх полей (`manager_email`, `manager_external_id`, `manager_name`), но лучше прислать **все три**:
- `manager_email` — приоритет №1 (стабильный, уникальный)
- `manager_external_id` — приоритет №2 (если email отсутствует)
- `manager_name` — fallback

### Таймзоны

Все даты — **ISO 8601 с явной таймзоной** (`+03:00`, `+05:00`, `Z` для UTC). Локальное время без таймзоны **не принимаем** (вернём 400).

---

## 5. Идемпотентность

Поле `external_id` — стабильный идентификатор встречи. Повторный webhook с тем же `external_id` и тем же `event` **не должен** приводить к дублированию данных на нашей стороне. Мы будем сравнивать и игнорировать повторы.

Это значит — если ваша система отправляет retry того же `meeting.started` — это безопасно, мы его не дублируем.

---

## 6. Retry policy на стороне календаря

### Коды ответов нашего сервера

| Код      | Значение                                                      | Действие календаря       |
|----------|---------------------------------------------------------------|--------------------------|
| `200`    | Принято и обработано                                          | success, не ретраить     |
| `202`    | Принято в очередь, обработается асинхронно                    | success, не ретраить     |
| `400`    | Невалидный payload (плохой JSON, отсутствуют обяз. поля)      | **не ретраить**, баг     |
| `401`    | Невалидный или отсутствует Bearer-токен                       | **не ретраить**, баг     |
| `404`    | Менеджер не найден на нашей стороне (не привязан к ноуту)     | ретраить через час       |
| `429`    | Слишком много запросов                                        | ретраить с backoff       |
| `5xx`    | Наш сервер сломался                                           | ретраить с backoff       |
| timeout  | Сеть/наш сервер не отвечает                                   | ретраить с backoff       |

### Backoff

Рекомендуемая стратегия:
- 1-я попытка: сразу
- 2-я: +5 секунд
- 3-я: +30 секунд
- 4-я: +5 минут
- 5-я: +30 минут
- 6-я: +2 часа

Желательно — **persistent queue** на стороне календаря, чтобы перезапуск/падение их сервиса не терял события.

## 7. Что мы делаем у себя

Чтобы не дёргали лишний раз:

- Эндпоинт `POST /webhook/calendar` с проверкой Bearer-токена и валидацией payload.
- Маппинг `manager_email` → `agent_id` (привязка email менеджера к его рабочему ноуту) — настраивается **админом один раз в дашборде**. От календаря — никаких настроек, просто шлём email/external_id.
- Идемпотентность по `external_id`.
- Логирование всех входящих webhook'ов (даже невалидных) — для разбора инцидентов.
- Привязка существующих голосовых сегментов (already-recorded аудио) к встрече по timestamp range — это даёт **нулевую задержку**: микрофон в нашей системе уже включён 24/7 через VAD, не нужно ничего "включать в момент начала".

---

## 8. Acceptance criteria

Считаем готовым когда:

1. ✅ В календаре настроен webhook на `meeting.started` и `meeting.ended`.
2. ✅ За один полный рабочий день (8 рабочих часов) все встречи менеджеров приходят к нам.
3. ✅ Для каждой встречи мы получаем оба события (`started` + `ended`) с корректными timestamp'ами.
4. ✅ Поле `external_id` уникальное в пределах одной встречи (один и тот же ID для start и end).
5. ✅ Bearer-токен передаётся в каждом запросе.
6. ✅ Retry в случае нашего downtime работает (можно проверить отключив наш сервис на 5 минут).
7. ✅ При повторной отправке (retry) одного и того же события дублей в нашей системе нет.
